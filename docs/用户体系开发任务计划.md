## 2. 目标与范围
### 2.1 目标
- 落地可上线的用户体系：注册、登录、刷新令牌、退出、个人信息、修改密码。
- 聊天接口改造为基于 Bearer Token 的用户识别，不再信任前端提交的 `user_id`。
- 支持会话归属校验，阻断跨用户访问同一 `session_id`。

### 2.2 In Scope
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/users/me`
- `PATCH /api/v1/users/me`
- `POST /api/v1/users/me/password`
- `POST /api/v1/chat` / `POST /api/v1/chat/stream` 鉴权改造
- 登录失败锁定（5 次 / 15 分钟）

### 2.3 Out of Scope
- 企业 SSO/OAuth
- 多租户与复杂 RBAC
- 独立管理后台 UI

## 3. 架构决策
- 认证机制：账号密码 + JWT（Access + Refresh）
- Refresh token：服务端哈希入库，轮换策略
- 迁移方案：SQL 脚本（`backend/migrations`）
- 前端令牌策略：Access 内存态，Refresh 存 localStorage
- 兼容策略：chat 接口保留 `user_id` 兼容字段 14 天，仅记录告警并忽略

## 4. 接口契约
### 4.1 新增接口
- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `GET /api/v1/users/me`
- `PATCH /api/v1/users/me`
- `POST /api/v1/users/me/password`

### 4.2 变更接口
- `POST /api/v1/chat`：请求体改为 `session_id/query/model_id`，`user_id` 仅兼容接收
- `POST /api/v1/chat/stream`：同上

### 4.3 错误码
- `401`：未认证、token 失效
- `403`：账号禁用、会话归属不匹配
- `409`：用户名/邮箱/手机号冲突
- `429`：登录失败次数过多触发锁定

## 8. 测试与验收用例
- 正常登录返回 access/refresh 双 token。
- 登录失败连续 5 次后返回 `429`。
- Access 过期后自动 refresh 并重试成功。
- `GET /users/me` 不返回 `password_hash`。
- 修改密码后旧密码立即失效。
- 未携带 token 调用 chat 返回 `401`。
- A 用户访问 B 用户 `session_id` 返回 `403`。
- SSE 协议 `meta/delta/done/error` 事件结构保持兼容。

## 9. 发布与回滚
- 发布顺序：后端先发（保留兼容），前端后发。
- 监控指标：登录成功率、401/403/429 比例、refresh 成功率、chat P95。
- 回滚规则：后端回滚前先关闭注册入口，防止新用户状态丢失。
