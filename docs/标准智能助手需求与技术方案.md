# 标准智能助手需求文档与技术方案（LangChain 1.2）

## 1. 文档信息
- 项目名称：标准智能助手（Web 单页问答版）
- 文档版本：v2.1
- 编写日期：2026-02-28
- 适用阶段：MVP（最小可用版本）

## 2. 项目背景与目标
### 2.1 项目背景
企业在研发、质量、采购、合规与项目交付过程中，经常需要查询标准信息（如国标、行标、团标、企业标准、国际标准）并理解条款含义。现状通常依赖人工翻找文档，效率低且容易漏检、误读版本。

### 2.2 项目目标
- 提供一个 Web 端标准智能问答助手，用户通过对话完成标准查询与条款解读。
- 支持条款级问答，并给出出处（标准编号 + 章节/条款）。
- 支持在对话中进行标准版本差异分析（新旧版变化摘要）。
- 在小规模部署条件下实现低成本、可维护、可追溯。

### 2.3 成功指标（MVP）
- 问答准确率 >= 80%（针对预置测试集）
- 带来源回答占比 >= 95%
- 平均响应时间 <= 3 秒，P95 <= 5 秒（含检索）
- 用户“有帮助”反馈 >= 80%

## 3. 需求范围
### 3.1 In Scope（本期范围）
- Web 单页对话式标准问答（浏览器访问）
- 后端 API 服务（供 Web 前端调用）
- 对话内标准事实查询（编号、名称、状态、发布日期、实施日期、归口单位等）
- 对话内条款问答与出处引用
- 对话内标准版本差异摘要（同一标准不同版本）
- 查询日志与反馈闭环

### 3.2 Out of Scope（本期不做）
- 标准全文下载与分发授权管理
- OCR 扫描版文档自动矫正（仅支持可解析文本）
- 多语言翻译（如 ISO 英文全文翻译）
- 独立列表页/标准详情页/独立比对页
- 自动生成合规报告（后续版本扩展）

## 4. 用户角色与核心用户故事
### 4.1 用户角色
- 研发工程师：查询技术指标与测试方法条款
- 质量/合规人员：确认标准有效性与版本状态
- 采购/项目经理：确认交付要求涉及的标准条款
- 知识管理员：维护标准库与更新策略

### 4.2 用户故事（MVP）
1. 作为研发工程师，我希望输入“某材料阻燃等级要求是什么”，系统返回对应标准条款和出处。
2. 作为质量人员，我希望通过一句话快速知道某标准是否现行有效以及最新版本号。
3. 作为项目经理，我希望在同一对话中比较某标准 2018 版与 2023 版关键变化。
4. 作为知识管理员，我希望看到用户问不到答案的问题，便于补充数据。

## 5. 功能需求
### 5.1 单页智能问答入口
- 前端提供单一对话页面（Chat UI）。
- 支持自然语言提问，包含标准事实查询、条款查询、版本比对。
- 支持多轮澄清（如“你问的是 GB/T 19001 还是 ISO 9001？”）。

### 5.2 对话内标准事实查询
- 可查询信息：标准编号、名称、状态、发布日期、实施日期、归口单位、适用行业。
- 返回结果需包含数据来源与更新时间。
- 不使用独立列表页承载结果，统一在对话回复中呈现。

### 5.3 条款级问答（RAG）
- 基于标准正文切片进行语义检索。
- 回答必须引用至少一个条款来源（标准编号 + 章/节/条）。
- 当依据不足时，不允许臆断，需返回“未找到充分依据”。

### 5.4 对话内版本比对
- 用户可在对话中发起比对（如“比较 GB/T xxxx-2018 和 2023 版差异”）。
- 输出新增条款、删除条款、关键变更摘要。
- 结果附带比对依据位置。

### 5.5 回答质量与边界控制
- 无引用不输出确定性结论。
- 低置信度时触发澄清问题而非强答。
- 遇到超出知识范围问题时明确说明边界。

### 5.6 反馈与运营
- 用户可标记“有帮助/无帮助”。
- 记录查询关键词、命中标准、引用条款、响应时长。
- 支持输出高频问题与失败问题榜单。

### 5.7 Web 前端展示需求
- 前端形态：仅 Web 端，不包含小程序与原生 App。
- 页面形态：仅单页问答助手（一个主页面）。
- 交互要求：
  - 支持流式输出（逐步显示答案）
  - 引用条款可点击并定位到对应章节
  - 对“依据不足”类回答提供澄清引导
- 兼容性要求：
  - 支持桌面端主流浏览器（Chrome、Edge）
  - 支持移动端浏览器基础可用（响应式布局）

## 6. 非功能需求
### 6.1 性能
- 普通问答平均响应 <= 3 秒，P95 <= 5 秒。
- 版本比对平均响应 <= 8 秒（后续可异步化扩展）。

### 6.2 可用性
- 可用性目标 >= 99.5%（MVP）。
- 检索服务异常时返回降级提示（无法检索、建议稍后重试）。

### 6.3 安全与合规
- 严格控制系统提示词、密钥和内部策略不可泄漏。
- 记录访问日志，便于审计。
- 对受版权或访问权限约束的标准正文进行访问控制。

### 6.4 可维护性
- 标准库支持增量更新（每日/每周任务）。
- 提示词、阈值、检索参数配置化。
- 具备可观测性（请求日志、模型trace、失败原因分类）。

## 7. 小规模技术选型（MVP建议）
### 7.1 选型原则
- 低运维成本，单机可部署。
- 尽量使用一个数据库承载结构化数据与向量索引。
- 保留后续扩展到中大型架构的接口边界。

### 7.2 技术栈
- 前端框架：React + Vite + TypeScript
- UI组件：Ant Design（小规模快速交付）
- 后端框架：FastAPI
- AI编排：LangChain 1.2 + LangGraph
- 模型：兼容 OpenAI API 的对话模型（主模型）+ 低成本回退模型
- 向量库：PostgreSQL + pgvector（小规模首选）
- 缓存：Redis（会话、热点查询缓存）
- 可观测：LangSmith + 应用日志
- 部署：Docker Compose

### 7.3 选型说明
- Web 端采用 React + Vite，快速交付单页对话体验。
- pgvector 可同时管理标准元数据和向量索引，降低系统复杂度。
- LangGraph 便于编排“意图识别 -> 条款检索/事实查询/版本比对 -> 答复组装”的状态流。
- Redis 可缓存高频问题，降低响应时间与模型成本。

## 8. 总体架构设计
### 8.1 逻辑架构
1. 用户在 Web 单页输入问题。
2. API 层完成鉴权、会话装载、限流。
3. Orchestrator 执行意图识别并路由：
   - 事实查询 -> Fact Retrieval Chain
   - 条款问答 -> RAG Chain
   - 版本比对 -> Compare Chain
4. 返回统一结果（answer + citations + structured_data），并写入日志与指标。

### 8.2 模块划分
- Web Frontend层：对话输入、流式展示、引用跳转、反馈提交
- API层：请求接入与响应封装
- Orchestrator层：LangGraph 状态编排
- Retrieval层：召回、重排、上下文组装
- Compare层：版本差异抽取与摘要
- Guardrails层：注入防护、低置信拒答、引用校验
- Observability层：日志、trace、指标

### 8.3 数据流
- 输入：`user_id + session_id + query`（来自 Web 前端）
- 中间：意图、检索结果、候选条款、引用片段、置信度
- 输出：`answer + citations + data + action`
- 存储：会话日志、检索日志、反馈日志

## 9. 关键技术方案
### 9.1 意图识别与路由
- 结构化意图定义：
  - `fact_query`
  - `clause_qa`
  - `version_compare`
  - `ambiguous_need_clarification`
- 策略：
  - 命中标准编号模式（如 `GB`, `GB/T`, `ISO`, `IEC`）优先走事实查询或条款链。
  - 未指定标准且问题泛化时，先澄清再检索。

### 9.2 对话内事实查询方案
- 先结构化过滤（编号、状态、年份）再语义召回。
- 返回字段标准化，避免同义字段混乱（如“现行有效”统一为 `active`）。
- 对话中附带来源与更新时间。

### 9.3 条款级 RAG 方案
- 文档处理：
  - 以“章-节-条”为主切片，保留层级ID。
  - 每个切片写入 `standard_code/version/chapter/clause` 元数据。
- 检索流程：
  - Query 重写（消歧、补全标准编号）
  - 向量召回 TopK=10
  - 规则过滤（同标准优先、同版本优先）
  - 重排 TopN=3
  - 生成回答并强制引用条款
- 约束：
  - 无引用不出结论。
  - 置信度低于阈值（建议 0.75）返回澄清或“无法确认”。

### 9.4 版本比对方案
- 输入：`standard_code + version_a + version_b`（由对话意图抽取）
- 方法：
  - 基于条款ID对齐（章/节/条）
  - LLM 仅做摘要，不做事实生成
  - 引用差异来源条款
- 输出结构：
  - `added_clauses[]`
  - `removed_clauses[]`
  - `changed_clauses[]`
  - `summary`

### 9.5 会话与缓存
- Redis 维护最近 10 轮对话上下文。
- 高频问题缓存（TTL 1-6 小时）。
- 超长会话自动摘要，降低 token 开销。

### 9.6 安全与质量控制
- Prompt 注入防护：忽略“泄露提示词/绕过规则”类请求。
- 输出校验：
  - 必须包含引用或明确声明依据不足。
  - 禁止输出无法追溯的绝对结论。
- 日志脱敏：用户身份、联系方式脱敏存储。

## 11. API 设计（MVP）
### 11.1 `POST /api/v1/chat`
- 入参：
  - `user_id`
  - `session_id`
  - `query`
- 出参：
  - `answer`
  - `citations[]`（标准编号、版本、条款）
  - `data`（结构化结果，可选）
  - `action`（`continue`/`clarify`）
  - `trace_id`

### 11.2 `POST /api/v1/feedback`
- 入参：`query_log_id`, `rating`, `comment`
- 出参：`ok`

### 11.3 `GET /api/v1/metrics`
- 出参：命中率、准确率、响应时长、反馈统计

## 12. 部署与运维方案（小规模）
### 12.1 部署方式
- Docker Compose 单机部署：
  - `web`（React 静态站点，Nginx 托管）
  - `app`（FastAPI + LangChain/LangGraph）
  - `postgres`（含 pgvector）
  - `redis`

### 12.2 配置管理
- `.env` 统一配置：
  - `MODEL_API_KEY`
  - `DB_URL`
  - `REDIS_URL`
  - `RAG_TOP_K`
  - `CONFIDENCE_THRESHOLD`
  - `CACHE_TTL_SECONDS`

### 12.3 告警策略
- 5xx 错误率 5 分钟窗口 > 3%
- 平均响应时长 > 5 秒
- 引用缺失率 > 5%

## 14. 验收标准
- 可完成：对话问答、标准事实查询、条款问答、版本比对。
- 回答可追溯：95% 以上输出含有效引用。
- 响应性能：满足第6章时延指标。
- 稳定性：可持续运行并具备基础告警能力。

## 15. 风险与应对
- 标准源数据质量不一致：建立字段映射与清洗规则。
- 版本映射困难：优先支持主流标准体系与明确版本号输入。
- 幻觉回答风险：强制引用校验，无依据不下结论。
- 成本压力：缓存热点问答、压缩上下文、分层模型调用。

## 16. 后续演进（MVP后）
- 增加混合检索（关键词 + 向量）提升召回。
- 加入标准变更订阅与提醒机制。
- 支持标准条款导出报告（按项目/产品维度）。
- 接入更多标准体系（如 ASTM、EN、JIS）与多语言检索。
