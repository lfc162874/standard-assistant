# 标准智能助手前后端详细开发计划（Web 单页问答）

## 1. 文档信息
- 项目：标准智能助手（Web 单页问答版）
- 文档类型：细节开发计划（前后端分轨）
- 版本：v1.0
- 编写日期：2026-02-28

## 2. 计划目标
- 把前端与后端开发任务拆成可执行步骤，避免职责不清。
- 明确每个阶段的输入、输出、依赖、验收标准。
- 保证开发顺序从易到难：先打通 `chat` 链路，再增强检索与质量控制，最后做版本比对与运营闭环。

## 3. 开发范围与边界
- 前端：仅 Web 单页问答，不做独立列表页、详情页、独立比对页。
- 后端：以 `POST /api/v1/chat` 为核心，支持对话内事实查询、条款问答、版本比对。
- 数据：标准元数据与条款正文入库，支持向量检索与引用追溯。

## 4. 里程碑总览（22天）
1. M0（Day 1）：工程初始化完成
2. M1（Day 2-4）：单页对话与 `chat` 接口联通
3. M2（Day 5-11）：RAG 核心链路上线
4. M3（Day 12-17）：引用校验与对话内版本比对
5. M4（Day 18-22）：反馈闭环、压测、灰度上线

## 5. 前后端协作约定
### 5.1 接口契约
- 契约先行：后端先给 OpenAPI 草案，前端按契约开发 mock。
- 版本规则：接口变更采用 `v1` 兼容策略，禁止无通知破坏性修改。
- 错误码统一：`code`、`message`、`trace_id` 必带。

### 5.2 联调机制
- 每天固定 1 次前后端站会（15分钟），同步阻塞问题。
- 每个阶段结束前进行一次“联调日”，集中处理契约偏差。
- 所有联调问题进入同一 Issue 列表，状态可追踪（open/in-progress/done）。

### 5.3 分支与提交流程
- 分支建议：
  - `codex/fe-*`：前端功能分支
  - `codex/be-*`：后端功能分支
  - `codex/integration-*`：联调分支
- 每个 PR 至少包含：
  - 变更说明
  - 自测结果
  - 风险点与回滚点

## 6. 前端详细开发计划

### FE-Step 0：项目脚手架与工程规范（Day 1）
目标：
- 搭建可维护的前端基础工程。

任务：
- 初始化 `React + Vite + TypeScript`。
- 集成 `Ant Design` 与全局样式变量。
- 建立目录：`src/pages`、`src/components`、`src/services`、`src/store`、`src/types`。
- 配置 ESLint、Prettier、基础 CI 检查。

产出：
- 前端工程可运行，基础页面可访问。

验收标准：
- `npm run dev` 与 `npm run build` 均成功。

---

### FE-Step 1：单页 Chat UI（Day 2-3）
目标：
- 完成单页问答助手界面和基础交互。

任务：
- 实现消息列表、输入框、发送按钮、清空会话。
- 实现消息类型：用户消息、助手消息、系统提示。
- 补齐状态：加载中、请求失败、重试。
- 适配响应式布局（桌面优先，移动端可用）。

产出：
- 可在浏览器进行连续对话交互（先接 mock 数据）。

验收标准：
- 连续发送 10 条消息界面稳定，无明显卡顿或错位。

---

### FE-Step 2：流式响应与会话管理（Day 4）
目标：
- 让回答“边生成边展示”，提升体感。

任务：
- 对接流式输出（SSE 或分块响应）。
- 支持中断显示与异常恢复。
- 引入会话标识 `session_id` 管理。
- 页面刷新后保留最近会话（本地存储即可）。

产出：
- 回答支持流式显示，会话可连续追问。

验收标准：
- 流式消息无乱码、无重复片段，异常时可提示重试。

---

### FE-Step 3：引用与证据展示（Day 8-10，依赖 BE-Step 3）
目标：
- 让用户看到答案依据并可定位条款。

任务：
- 在回答下方展示 `citations`（标准编号、版本、章节/条款）。
- 支持点击引用后在侧边抽屉展示条款正文片段。
- 对“依据不足”回答显示澄清提示卡片。

产出：
- 用户可直接查看回答证据。

验收标准：
- 95% 以上回答可正确渲染引用块（以联调样例为准）。

---

### FE-Step 4：对话内版本比对展示（Day 14-16，依赖 BE-Step 5）
目标：
- 在同一聊天窗口展示版本差异结果。

任务：
- 支持 `compare` 类型消息渲染。
- 结构化展示新增/删除/修改条款。
- 摘要与差异条款引用可折叠展开。

产出：
- 用户通过自然语言触发比对后，可在聊天窗口直接看结果。

验收标准：
- 10 组样例中，结果渲染完整率 100%。

---

### FE-Step 5：反馈与埋点（Day 18-19）
目标：
- 建立前端可观测与反馈入口。

任务：
- 每条助手回答提供“有帮助/无帮助”反馈按钮。
- 上报基础埋点：提问次数、失败次数、重试次数、反馈结果。
- 对接 `POST /api/v1/feedback`。

产出：
- 反馈链路可用，前端行为可统计。

验收标准：
- 反馈提交成功率 >= 99%（联调环境）。

---

### FE-Step 6：发布前优化（Day 20-22）
目标：
- 保障上线体验与稳定性。

任务：
- 首屏性能优化（懒加载、打包优化）。
- 统一错误提示与降级文案。
- 浏览器兼容验证（Chrome、Edge，移动端浏览器基础验证）。

产出：
- 可发布前端构建产物。

验收标准：
- Lighthouse 基础可用评分达标（内部阈值自定）。

## 7. 后端详细开发计划

### BE-Step 0：服务骨架与基础设施（Day 1）
目标：
- 建立可扩展后端工程。

任务：
- 初始化 FastAPI 项目结构（`api`、`services`、`chains`、`repositories`）。
- 接入配置管理（`.env`）与结构化日志。
- 提供健康检查接口 `/health`。
- 配置 Docker Compose（`app/postgres/redis`）。

产出：
- 后端服务可本地运行并可被前端调用。

验收标准：
- `GET /health` 稳定返回，容器启动无报错。

---

### BE-Step 1：`chat` 接口骨架与会话日志（Day 2-4）
目标：
- 先打通对话接口，哪怕先返回规则模板答案。

任务：
- 实现 `POST /api/v1/chat` 入参校验与返回结构。
- 建立 `sessions`、`query_logs` 基础表。
- 记录请求日志、耗时、`trace_id`。
- 返回统一错误码与异常处理中间件。

产出：
- 前端可调用 `chat` 并拿到稳定结构响应。

验收标准：
- 接口稳定性通过基本压力测试（并发 20 无异常）。

---

### BE-Step 2：数据层建设（Day 5-7）
目标：
- 准备检索所需数据与索引。

任务：
- 建立 `standards`、`standard_clauses` 表与索引。
- 完成标准数据导入脚本（清洗、切片、入库）。
- 接入 pgvector 扩展与向量字段。

产出：
- 可检索标准数据集（建议首批 >= 50份标准文档）。

验收标准：
- 数据抽检通过，切片结构正确率 >= 95%。

---

### BE-Step 3：RAG 核心链路（Day 8-11）
目标：
- 实现真实的条款级问答能力。

任务：
- 实现 Query 重写、向量召回、重排、答案生成。
- 统一输出 `citations` 字段。
- 通过 LangGraph 编排 `fact_query/clause_qa` 路由。
- 引入 Redis 会话上下文缓存（最近10轮）。

产出：
- `chat` 支持条款问答并附证据来源。

验收标准：
- 测试集准确率 >= 80%，带引用回答占比 >= 95%。

---

### BE-Step 4：回答质量控制（Day 12-13）
目标：
- 控制幻觉，提升可信度。

任务：
- 增加输出校验：无引用不输出确定结论。
- 增加低置信阈值策略与澄清流程。
- 增加注入攻击防护与敏感输出拦截。

产出：
- 回答质量控制策略生效。

验收标准：
- 引用缺失率 <= 5%，幻觉率较上阶段明显下降。

---

### BE-Step 5：对话内版本比对（Day 14-17）
目标：
- 支持聊天内新旧标准差异分析。

任务：
- 实现比对逻辑：条款对齐、差异识别、摘要生成。
- 在 `chat` 路由中增加 `version_compare` 分支。
- 返回结构化差异数据（added/removed/changed/summary）。

产出：
- 用户可在同一聊天会话内完成版本比对请求。

验收标准：
- 10 组样例中，关键变化识别正确率 >= 80%。

---

### BE-Step 6：反馈与指标（Day 18-19）
目标：
- 建立运营闭环与可观测能力。

任务：
- 开发 `POST /api/v1/feedback`、`GET /api/v1/metrics`。
- 聚合指标：准确率、响应时长、无帮助率、引用缺失率。
- 接入 LangSmith 追踪关键链路节点。

产出：
- 可按天查看核心效果指标。

验收标准：
- 指标统计口径一致，可导出日报数据。

---

### BE-Step 7：上线前加固（Day 20-22）
目标：
- 保证可上线、可回滚、可监控。

任务：
- 压测与性能调优（慢查询优化、缓存命中提升）。
- 完成告警配置（5xx、延迟、引用缺失率）。
- 制定回滚预案（模型异常、数据库异常）。
- 完成上线检查清单。

产出：
- 灰度上线版本与回滚方案。

验收标准：
- 压测达标，演练通过，告警可触达。

## 8. 关键依赖关系（前后端）
- FE-Step 2 依赖 BE-Step 1：流式接口协议稳定后再接入。
- FE-Step 3 依赖 BE-Step 3：`citations` 结构固定后再渲染。
- FE-Step 4 依赖 BE-Step 5：版本比对返回结构稳定后再渲染。
- FE-Step 5 依赖 BE-Step 6：反馈接口与指标埋点对齐。

## 9. 风险与缓冲
- 风险：标准源数据质量波动
  - 缓冲：BE-Step 2 增加抽检脚本与字段映射校验
- 风险：流式协议在不同环境行为不一致
  - 缓冲：保留非流式降级模式
- 风险：版本比对准确率不达标
  - 缓冲：先限制支持范围（主流标准+明确版本号）

## 10. 最终交付物
- Web 单页智能问答前端（可部署）
- 后端 API 服务（含 RAG、比对、反馈、指标）
- 标准数据导入脚本与更新说明
- 接口文档（OpenAPI）
- 测试报告、压测报告、上线回滚方案
