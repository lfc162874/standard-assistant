# 标准智能助手开发计划（循序渐进）

## 1. 文档信息
- 项目：标准智能助手（Web 单页问答版）
- 文档类型：开发计划
- 版本：v1.1
- 编写日期：2026-02-28

## 2. 开发原则
- 先通路、后优化：先把最短问答链路跑通，再补质量与性能。
- 先简单、后复杂：先做对话问答 MVP，再做引用校验，最后做版本比对增强。
- 每一步都可验收：每阶段必须有可演示结果，避免“做了很多但不能用”。
- 小步快跑：每一步控制在 1-3 天可完成范围。

## 3. 总体里程碑（由易到难）
1. 基础环境与项目骨架
2. 单页对话助手与 `chat` 接口打通
3. 条款数据入库与 RAG 检索
4. 引用溯源与回答质量控制
5. 对话内版本比对能力
6. 反馈闭环与运营指标
7. 稳定性、性能与上线

## 4. 详细开发步骤

### Step 0：项目初始化（第 1 天）
目标：
- 建立可运行的前后端仓库结构与开发规范。

任务：
- 创建目录结构：`web/`、`app/`、`docs/`、`scripts/`。
- 初始化后端 FastAPI 项目与前端 React + Vite 项目。
- 配置 `.env.example`、日志格式、基础 README。
- 建立 Docker Compose（`web`、`app`、`postgres`、`redis`）。

产出：
- 本地一键启动命令可运行（例如 `docker compose up`）。
- 前端可打开问答页面，后端 `/health` 可访问。

验收标准：
- 团队成员按 README 可在 30 分钟内完成本地启动。

---

### Step 1：单页对话 MVP（第 2-4 天）
目标：
- 先完成“能问能答”的最小闭环。

任务：
- 前端实现单页 Chat UI（输入框、消息区、加载态、错误态）。
- 后端开发 `POST /api/v1/chat`（先返回规则化模拟答案）。
- 打通会话 ID 与消息日志落库。
- 打通流式返回（SSE 或分块响应）。

产出：
- 用户可以在一个页面连续提问并收到回答。

验收标准：
- 连续 10 轮对话不中断。
- 基础交互无阻塞，平均响应 <= 2 秒（模拟数据阶段）。

---

### Step 2：条款数据入库与切片（第 5-7 天）
目标：
- 为 RAG 做准备，构建可检索标准条款数据层。

任务：
- 设计 `standards`、`standard_clauses` 表结构。
- 编写切片规则：按“章-节-条”切分，保留层级路径。
- 编写导入脚本：正文解析 -> 清洗 -> 切片 -> 入库。
- 抽样校验切片质量（标题丢失、断句错误、重叠率）。

产出：
- 至少 50 份标准正文完成可检索化处理。

验收标准：
- 抽样 100 条切片中，结构正确率 >= 95%。

---

### Step 3：RAG 问答核心链路（第 8-11 天）
目标：
- 实现“问条款 -> 回答 + 引用”的核心能力。

任务：
- 接入 embedding 生成与 pgvector 索引。
- 实现检索链路：Query 重写 -> 召回 -> 重排 -> 生成。
- 将 `chat` 接口切换为真实 RAG 回答。
- 接入 Redis 会话缓存（最近 10 轮）。

产出：
- Web 单页中可针对标准条款提问并得到可用回答。

验收标准：
- 回答准确率（测试集）>= 80%。
- 带引用回答占比 >= 95%。

---

### Step 4：引用溯源与回答质量控制（第 12-14 天）
目标：
- 避免“看似正确但无依据”的回答。

任务：
- 增加输出校验：无引用则拒绝输出确定性结论。
- 增加低置信策略：触发澄清问题而非强答。
- 前端支持引用点击定位到条款位置。
- 记录每次回答引用来源到 `query_logs`。

产出：
- 回答可追溯，用户可点击引用查看来源条款。

验收标准：
- 引用缺失率 <= 5%。
- 幻觉样本比例较 Step 3 显著下降。

---

### Step 5：对话内版本比对（第 15-17 天）
目标：
- 在同一对话中支持新旧版标准差异分析。

任务：
- 实现比对链路：条款对齐 -> 差异识别 -> 摘要生成。
- 在 `chat` 内识别比对意图并触发比对流程。
- 返回新增/删除/修改条款及摘要，并附引用。

产出：
- 用户通过自然语言在聊天中完成版本比对。

验收标准：
- 10 组人工验证样本中，关键变化识别正确率 >= 80%。

---

### Step 6：反馈闭环与运营指标（第 18-19 天）
目标：
- 建立“可持续优化”的数据闭环。

任务：
- 开发 `POST /api/v1/feedback`。
- 建立指标汇总：准确率、响应时长、无帮助率、引用缺失率。
- 增加失败问题榜单（便于知识补充）。
- 完成基础运营看板（可先用简单表格页）。

产出：
- 运营可查看系统效果并定位短板。

验收标准：
- 能按天查看核心指标并导出 CSV。

---

### Step 7：上线前收尾（第 20-22 天）
目标：
- 保障上线稳定性与可维护性。

任务：
- 完成接口与链路压测。
- 增加告警：5xx、慢查询、引用缺失率。
- 完成回滚方案与应急预案（模型不可用、数据库异常）。
- 完成上线检查清单（配置、密钥、备份、监控）。

产出：
- 可灰度上线的发布版本。

验收标准：
- 压测达标，关键告警有效，回滚演练通过。

## 5. 每周节奏建议
- 周一：确定本周目标与验收标准。
- 周二至周四：开发与联调。
- 周五：测试、复盘、文档更新。

## 6. 测试策略（建议同时推进）
- 单元测试：切片逻辑、字段映射、比对逻辑。
- 集成测试：检索链路、`chat` 接口、数据库与缓存联调。
- 回归测试：固定问题集（每次迭代跑一遍）。
- 人工验收：产品与业务同学抽检 20-50 条典型问题。

## 7. 交付清单
- 需求与技术方案文档
- 开发计划文档（本文）
- 前后端可运行代码
- 数据导入与更新脚本
- API 文档（OpenAPI）
- 测试报告与上线记录

## 8. 风险控制
- 风险：标准源数据更新不及时
  - 应对：建立每周增量更新任务与数据版本号
- 风险：条款切片质量差导致检索偏差
  - 应对：增加切片质量抽检与自动校验规则
- 风险：模型成本上升
  - 应对：缓存热点问题、限制上下文长度、分层模型策略

## 9. 一句话执行顺序
先把“单页问答能稳定跑通”做成，再把“条款回答有引用、可追溯”做强，最后把“版本比对和运营闭环”做完整。
